## Root Cause Analysis: CPU 40% の壁

### ランタイム環境 (bench-target pod 内部調査)

```
nproc:                 2 (JVMから見えるCPUコア数 = ホスト物理コア)
cgroup cpu.cfs_quota:  112000
cgroup cpu.cfs_period: 100000
→ 実効CPU制限:        1.12 vCore (112000/100000)
Memory limit:          4 GB (cgroup)
Heap Max (JVM):        2,048 MB
```

**確認: bench-target は v-cores=1 でデプロイ済み (cgroup 実測 1.12 vCore)。mule-perf も同様に 1 vCore。**

### Mule 4 スレッドプール構成 (実測)

| スレッドグループ | スレッド数 | 役割 |
|----------------|-----------|------|
| `[MuleRuntime].u` (UBER) | **2** | CPU_INTENSIVE処理 (DataWeave変換等) |
| `[MuleRuntime]._` (CPU_LITE) | **2** | 軽量処理 (ルーティング、ログ等) |
| `[MuleRuntime].t` (I/O) | **1** | ブロッキングI/O |
| `http.listener.0` (NIO) | **4** | HTTP受信 (Nettyイベントループ) |
| その他 (C3P0, servo等) | ~60 | DB接続プール、モニタリング等 |

### なぜ CPU 40% で止まるのか

Mule 4 のスレッドプールサイズは `Runtime.availableProcessors()` に基づく:
- CPU_INTENSIVE (UBER): `max(2, cores)` = **2スレッド**
- nproc=2 (cgroupの1.12 vCoreではなく、ホストの物理コア数を返す)

**DataWeave変換は CPU_INTENSIVE カテゴリで処理される。**
→ 同時に DataWeave を実行できるのは **最大2スレッド** のみ。

2スレッドが1.12コアのうち約40%を消費 → **これがCPU 40%の壁の正体**。
コンカレンシーをいくら上げても、DataWeave処理は2スレッドのキューに並ぶだけ。

### T1-A Echo がCPU 97%まで使える理由

Echo は DataWeave の `now()` のみで CPU_LITE カテゴリ。
NIO の `http.listener` (4スレッド) + CPU_LITE (2スレッド) で処理されるため、
より多くのスレッドが利用可能 → CPU 97% まで活用できる。

ただし c=100 以上で突然 CPU 40% に落ちるのは、NIO スレッドのバックプレッシャーが効いて
CPU_INTENSIVE にフォールバックしている可能性がある。

### チューニングの余地

`scheduler-pools.conf` (または JVM システムプロパティ) でスレッドプールサイズを変更可能:
```
# org.mule.runtime.scheduler.cpuIntensive.threadPool.size=8
# org.mule.runtime.scheduler.cpuLight.threadPool.size=8
```
→ スレッド数を増やせば CPU 利用率が上がり、RPS 向上が期待できる。
ただし 1 vCore 環境ではコンテキストスイッチのオーバーヘッドで逆効果になる可能性もある。

## 0.5 vCore の場合の予測

### 環境の違い

| 項目 | 1 vCore (実測) | 0.5 vCore (予測) |
|------|--------------|-----------------|
| cgroup quota | 112000 (1.12コア) | ~56000 (0.56コア) |
| nproc (JVM) | 2 | 2 (変わらない※) |
| UBER スレッド | 2 | 2 (nproc依存) |
| CPU_LITE スレッド | 2 | 2 |
| NIO スレッド | 4 | 4 |
| Heap Max | 2,048 MB | ~1,024 MB (半減) |

※ nproc はホストの物理コア数を返す。cgroup の quota が変わっても nproc=2 のまま。

### 予測パフォーマンス

スレッドプールサイズは変わらないが、各スレッドが使えるCPU時間が半分になる:

| シナリオ | 1 vCore 実測 | 0.5 vCore 予測 | 根拠 |
|---------|-------------|---------------|------|
| **T1-A Echo** | 2,061 RPS | **~1,000 RPS** | CPU律速 (97%→48%) |
| **T1-B Small DW** | 650 RPS | **~325 RPS** | UBER 2スレッドのCPU時間が半減 |
| **T1-C Heavy DW** | 160 RPS | **~80 RPS** | UBER 2スレッドのCPU時間が半減 |

### 詳細な予測根拠

**T1-A Echo (CPU_LITE + NIO)**:
- 1 vCoreで CPU 97% → ほぼCPU律速
- CPU時間が半分 → RPS もほぼ半分 (~1,000)
- レイテンシ: c=50 で avg ~50ms (24ms の約2倍)

**T1-B Transform Small (CPU_INTENSIVE)**:
- 1 vCoreで 2スレッドが CPU 40% 使用
- 0.5 vCoreでも 2スレッドだが、CPU時間半減 → CPU ~40% のまま実効処理量は半分
- RPS ~325、c=200 で avg ~600ms

**T1-C Transform Heavy (CPU_INTENSIVE)**:
- DataWeave 1リクエスト ~11ms (c=1) → 0.5 vCore では ~22ms に
- 2スレッド × 1000ms/22ms = ~91 RPS (実測160の約57%)
- 実際には I/O待ちやGCのオーバーヘッドで **~80 RPS** 程度と予測

### 0.5 vCore での注意点

1. **Heap 削減の影響**: 1,024 MB になると GC 頻度が増加し、上記予測よりさらに低下する可能性
2. **起動時間**: Mule ランタイムの起動自体が遅くなる (CPU制限)
3. **過負荷耐性は維持**: Mule 4 のバックプレッシャー機構はvCoreに依存しない
4. **スレッドプール変わらず**: nproc=2 のままなので、スレッドプール制約は同一

---

## Key Findings

1. **持続負荷での真の性能値**: Echo 2,061 RPS / 軽量DW 650 RPS / 重量DW 160 RPS
2. **CPU 40%の壁の正体**: UBER (CPU_INTENSIVE) スレッドプール = 2スレッド (nproc=2 に基づく)
3. **メモリは余裕**: Heap最大529MB / 2,048MB (26%)。メモリはボトルネックではない
4. **過負荷時の挙動**: Echo c=100以上でCPU 40%に落ちてRPS半減 (スレッド競合コスト)
5. **全テスト エラー率 0%**: 180秒 × c=500 でもMule 4は一切エラーを出さない
6. **60秒テストの罠**: 短時間テストではバースト性能が出て、実際の持続性能を過大評価しうる
7. **0.5 vCore では概ね半減**: スレッドプール構成は変わらないが、CPU時間半減で RPS ほぼ半分
